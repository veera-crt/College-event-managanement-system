Sprint Retrospective

What went well	What went poorly	What ideas do you have	How should we take action

Certificate Drive Integration: We successfully implemented a Google Drive link-based upload system. The backend middleware can now fetch the link and distribute the certificates to all eligible students in a matter of seconds.	Bulk Folder Uploading Failed: Initially, uploading certificates directly from a local folder caused severe performance bottlenecks. The server struggled and timed out when trying to process bulk files all at once.	Cloud-Based File Handling: Instead of forcing our local server to process heavy files simultaneously, we should rely on cloud storage URLs for any future bulk data transfers.	Standardize Cloud Uploads: For the next sprint, update the API guidelines to ensure any bulk media (like gallery photos) is submitted via a cloud link rather than direct file uploads to save server bandwidth.

QR-Ready Unique Ticketing: We successfully updated the backend to generate a brand new, unique Ticket ID for every single user booking. This perfectly sets up the system to track exact ticket details via QR codes.	Ticket Recognition Failure: After the initial cultural registration build, the system could not recognize or validate individual tickets because bookings lacked a dedicated unique identifier.	Server-Side ID Generation: We realized that transactional records like event tickets must always have a unique string or UUID generated strictly at the server level to ensure foolproof tracking.	QR Validation Implementation: Now that every booking has a unique server-generated ID, our immediate action for the next phase is to tie these IDs to a QR scanner so organizers can easily validate physical entry at the event gates.
